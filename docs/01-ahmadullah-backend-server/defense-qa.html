Backend Defense Q&A

سوال و جواب کامل برای دفاع احمدالله

این سوال‌ها را مثل متن حفظ نکنید. اول جواب کوتاه را تمرین کنید، بعد اگر استاد بیشتر پرسید، جواب عمیق را توضیح بدهید و در صورت نیاز با لینک Cursor فایل مربوطه را باز کنید.

گروه ۱: معماری و ساختار پروژه

۱. Backend این پروژه دقیقاً چه مسؤولیتی دارد؟

جواب کوتاه: Backend قوانین اصلی سیستم، امنیت، دیتابیس و APIها را مدیریت می‌کند.

جواب عمیق: فرانت‌اند request می‌فرستد، اما Backend تصمیم می‌گیرد کاربر اجازه دارد یا نه، input درست است یا نه، کدام SQL اجرا شود، audit log نوشته شود یا نه، و جواب به چه شکل برگردد.

where in code?

۲. چرا modular monolith انتخاب شده است؟

جواب کوتاه: چون برای سیستم واحد پوهنتونی ساده‌تر و قابل مدیریت‌تر است.

جواب عمیق: همه featureها در یک سرور هستند، پس deploy ساده است؛ اما هر feature route جدا دارد، پس کد منظم می‌ماند. این انتخاب برای تیم کوچک و پروژه پوهنتونی منطقی است.

where in code?

۳. سرور از کدام فایل شروع می‌شود؟

جواب کوتاه: از server/src/server.ts.

جواب عمیق: این فایل createApp() را صدا می‌زند، سرور را با app.listen اجرا می‌کند، و برای shutdown درست، pool دیتابیس را می‌بندد.

where in code?

۴. فایل app.ts چه کار می‌کند؟

جواب کوتاه: Express app را می‌سازد و middlewareها و routeها را وصل می‌کند.

جواب عمیق: در این فایل session store، CORS، Helmet، JSON parser، CSRF، routeهای featureها، static production frontend، not found و error handler تنظیم می‌شود.

where in code?

گروه ۲: Middleware و Request Lifecycle

۵. Middleware چیست؟

جواب کوتاه: کدی است که قبل یا بعد از route روی request اجرا می‌شود.

جواب عمیق: Middleware می‌تواند request را log کند، body را parse کند، session را بخواند، امنیت را بررسی کند، یا خطا را به جواب JSON تبدیل کند.

where in code?

۶. مسیر یک request در سرور چیست؟

جواب کوتاه: browser → Express app → middleware → router → دیتابیس → response.

جواب عمیق: اول browser از API درخواست می‌فرستد. Express middlewareهای عمومی را اجرا می‌کند. سپس request به route مربوطه می‌رود. route SQL یا helper را اجرا می‌کند و در آخر JSON برمی‌گرداند.

۷. چرا CORS لازم است؟

جواب کوتاه: تا browser فقط از origin مجاز به API درخواست بدهد.

جواب عمیق: چون frontend و backend ممکن است روی portهای جدا اجرا شوند. CORS مشخص می‌کند کدام origin اجازه دارد cookie/session را همراه request بفرستد.

where in code?

۸. Helmet برای چیست؟

جواب کوتاه: headerهای امنیتی HTTP را اضافه می‌کند.

جواب عمیق: Helmet کمک می‌کند بعضی خطرات عمومی وب مثل clickjacking یا headerهای ناامن کاهش یابد. در development، CSP خاموش شده تا Vite راحت‌تر کار کند.

where in code?

گروه ۳: Authentication, Session و Password

۹. Login چگونه کار می‌کند؟

جواب کوتاه: identifier و password validate می‌شود، user پیدا می‌شود، password بررسی می‌شود، session ساخته می‌شود.

جواب عمیق: route اول input را با Zod چک می‌کند، user را با email یا username پیدا می‌کند، lock و status را بررسی می‌کند، password را با Argon2 verify می‌کند، session را regenerate می‌کند، userId و activeAssignmentId را ذخیره می‌کند و CSRF token برمی‌گرداند.

where in code?

۱۰. چرا Argon2 استفاده شده است؟

جواب کوتاه: چون برای password hashing امن و مناسب است.

جواب عمیق: پسورد خام ذخیره نمی‌شود. فقط hash ذخیره می‌شود. هنگام login، پسورد واردشده با hash مقایسه می‌شود. Argon2 برای سخت‌تر کردن حمله به password hashها طراحی شده است.

where in code?

۱۱. اگر password غلط باشد چه می‌شود؟

جواب کوتاه: تعداد تلاش ناکام زیاد می‌شود و بعد از چند بار حساب موقتاً قفل می‌شود.

جواب عمیق: کد failed_login_attempts را افزایش می‌دهد. اگر تعداد به ۵ برسد، locked_until برای ۱۵ دقیقه تنظیم می‌شود. این در برابر brute force کمک می‌کند.

where in code?

۱۲. چرا session بعد از login regenerate می‌شود؟

جواب کوتاه: برای امنیت بیشتر و جلوگیری از session fixation.

جواب عمیق: اگر session قدیمی قبل از login وجود داشته باشد، بعد از login session تازه ساخته می‌شود تا attacker نتواند از session قبلی سوءاستفاده کند.

where in code?

۱۳. active assignment چیست؟

جواب کوتاه: موقف یا وظیفه فعلی کاربر در سیستم است.

جواب عمیق: سیستم اداری فقط user ندارد؛ هر user به یک person و assignment وصل است. active assignment مشخص می‌کند کاربر در حال حاضر از طرف کدام position و unit عمل می‌کند.

where in code?

۱۴. Logout چگونه کار می‌کند؟

جواب کوتاه: audit log نوشته می‌شود و session destroy می‌شود.

جواب عمیق: وقتی user logout می‌کند، سیستم action را ثبت می‌کند و session را از بین می‌برد تا cookie دیگر دسترسی نداشته باشد.

where in code?

گروه ۴: CSRF، Roles و امنیت

۱۵. CSRF چیست؟

جواب کوتاه: حمله‌ای است که یک سایت دیگر از طرف کاربر login شده request می‌فرستد.

جواب عمیق: چون browser cookie را خودکار می‌فرستد، یک سایت مخرب می‌تواند تلاش کند request خطرناک بفرستد. CSRF token ثابت می‌کند request از frontend خود سیستم آمده است.

where in code?

۱۶. کدام methodها CSRF نیاز دارند؟

جواب کوتاه: POST، PUT، PATCH و DELETE.

جواب عمیق: این methodها معلومات را تغییر می‌دهند. GET معمولاً فقط خواندن است، پس در این guard به عنوان unsafe method حساب نشده است.

where in code?

۱۷. چرا route login از CSRF معاف است؟

جواب کوتاه: چون کاربر قبل از login هنوز CSRF token ندارد.

جواب عمیق: بعد از login، token داخل session ساخته و به client داده می‌شود. پس خود login و forgot/reset password باید بدون token قابل اجرا باشند.

where in code?

۱۸. requireAuth چه کار می‌کند؟

جواب کوتاه: بررسی می‌کند کاربر login است و حسابش active است.

جواب عمیق: اگر session userId نداشته باشد، unauthorized می‌دهد. اگر user حذف یا inactive باشد، session destroy می‌شود. سپس roleها خوانده می‌شود و در response.locals قرار می‌گیرد.

where in code?

۱۹. requireAnyRole چه کار می‌کند؟

جواب کوتاه: بررسی می‌کند user یکی از roleهای مجاز را دارد.

جواب عمیق: مثلاً routeهای admin فقط برای system_admin یا admin_staff باز است. اگر user role لازم نداشته باشد، forbidden برمی‌گردد.

where in code?

گروه ۵: Database، Migration و Seed

۲۰. MySQL pool چیست؟

جواب کوتاه: مجموعه‌ای از connectionهای قابل استفاده مجدد به دیتابیس است.

جواب عمیق: ساختن connection برای هر request کند است. pool چند connection نگه می‌دارد و requestها از آنها استفاده می‌کنند. در production تعداد connection بیشتر تنظیم شده است.

where in code?

۲۱. env.ts چه مشکلی را حل می‌کند؟

جواب کوتاه: تنظیمات محیط را validate و centralize می‌کند.

جواب عمیق: با Zod مشخص می‌شود PORT عدد مثبت باشد، APP_ORIGIN URL باشد، SESSION_SECRET حداقل طول داشته باشد و defaultهای local development وجود داشته باشد.

where in code?

۲۲. Migration چیست؟

جواب کوتاه: فایل‌هایی که ساختار دیتابیس را مرحله‌به‌مرحله می‌سازند.

جواب عمیق: migration باعث می‌شود همه اعضای تیم و سیستم deploy یک schema یکسان داشته باشند. script جدول schema_migrations می‌سازد تا بداند کدام migration قبلاً اجرا شده است.

where in code?

۲۳. Seed چیست؟

جواب کوتاه: داده‌های اولیه لازم برای شروع سیستم را وارد دیتابیس می‌کند.

جواب عمیق: seed حساب admin، role، organization، unit، position، assignment و داده‌های پایه document reference را می‌سازد تا بعد از نصب بتوان login و کار کرد.

where in code?

۲۴. جدول sessions برای چیست؟

جواب کوتاه: sessionهای login کاربران را ذخیره می‌کند.

جواب عمیق: Express session با MySQLStore به جدول sessions وصل است. cookie فقط session id را نگه می‌دارد؛ data اصلی session در دیتابیس ذخیره می‌شود.

where in code?

گروه ۶: Validation، Errors و Audit

۲۵. چرا Zod استفاده شده است؟

جواب کوتاه: برای validate کردن inputهای request.

جواب عمیق: اگر input نادرست باشد، قبل از اجرای منطق اصلی route خطا داده می‌شود. این کار هم امنیت را بهتر می‌کند و هم خطاها را قابل پیش‌بینی می‌سازد.

where in code?

۲۶. asyncHandler چه کار می‌کند؟

جواب کوتاه: خطاهای routeهای async را به error handler می‌فرستد.

جواب عمیق: اگر promise reject شود، بدون try/catch تکراری، خطا به next داده می‌شود و Express error middleware آن را مدیریت می‌کند.

where in code?

۲۷. AppError چیست؟

جواب کوتاه: خطای قابل پیش‌بینی با status code و code مشخص است.

جواب عمیق: به جای error عمومی، AppError مشخص می‌کند HTTP status و error code چه باشد؛ مثل unauthorized، forbidden یا invalid_credentials.

where in code?

۲۸. errorHandler چند نوع خطا را مدیریت می‌کند؟

جواب کوتاه: AppError، ZodError و خطای ناشناخته.

جواب عمیق: AppError با status خودش برمی‌گردد. ZodError به validation_failed با status 422 تبدیل می‌شود. خطای ناشناخته log می‌شود و status 500 می‌گیرد.

where in code?

۲۹. Audit log چرا مهم است؟

جواب کوتاه: برای پیگیری کارهای مهم کاربران.

جواب عمیق: سیستم اداری باید traceable باشد. audit log ثبت می‌کند چه کسی، با کدام assignment، چه actionی را روی کدام entity انجام داده است.

where in code?

گروه ۷: سوال‌های عملی و Demo

۳۰. اگر استاد بگوید login را در کد نشان بده، چه می‌کنید؟

جواب کوتاه: فایل auth.routes.ts را باز می‌کنم و route POST /login را نشان می‌دهم.

جواب عمیق: از line 260 شروع می‌کنم؛ validate input، پیدا کردن user، password verification، lockout، session و audit را مرحله‌به‌مرحله توضیح می‌دهم.

open login code

۳۱. اگر استاد بپرسد APIها کجا register شده‌اند؟

جواب کوتاه: در server/src/app.ts.

جواب عمیق: از line 93 به بعد، prefixهای routeها مثل /api/auth، /api/documents و /api/admin به routerها وصل شده‌اند.

open route registration

۳۲. اگر استاد بپرسد دیتابیس از کجا وصل می‌شود؟

جواب کوتاه: از server/src/db/mysql.ts.

جواب عمیق: این فایل با استفاده از env config یک MySQL connection pool می‌سازد و همه routeها از همین pool برای query استفاده می‌کنند.

open mysql pool

۳۳. اگر استاد بپرسد چطور پروژه را local اجرا کنیم؟

جواب کوتاه: MySQL را اجرا می‌کنیم، دیتابیس را create/migrate/seed می‌کنیم، بعد npm run dev.

npm run db:create npm run db:migrate npm run db:seed npm run dev

۳۴. اگر استاد بپرسد خطاها چطور به frontend می‌رسند؟

جواب کوتاه: به شکل JSON استاندارد با object خطا.

جواب عمیق: errorHandler خطا را به ساختار { error: { code, message, details } } تبدیل می‌کند تا frontend بتواند پیام مناسب نشان دهد.

open error handler

۳۵. اگر استاد بپرسد چرا از session به جای JWT استفاده شده؟

جواب کوتاه: چون سیستم داخلی پوهنتونی است و session server-side کنترول و revoke آسان‌تر دارد.

جواب عمیق: session در MySQL ذخیره می‌شود، cookie فقط شناسه دارد، و سرور هر request را با دیتابیس/session store بررسی می‌کند. برای سیستم closed university این مدل ساده و قابل کنترول است.
