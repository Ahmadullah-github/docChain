نقشه ذهنی سریع

Backend در DocChain مثل اداره مرکزی سیستم است. فرانت‌اند فقط صفحه و button را نشان می‌دهد؛ اما تصمیم اصلی در سرور گرفته می‌شود: آیا کاربر login است؟ آیا role مناسب دارد؟ آیا input درست است؟ آیا سند یا کاربر در دیتابیس وجود دارد؟ آیا باید audit log نوشته شود؟

Browser / React UI -> client/src/api/* -> Express app in server/src/app.ts -> Middleware chain -> Feature router -> MySQL query / transaction -> JSON response: { data: ... } or { error: ... }

چطور به استاد توضیح بدهید؟

«در این پروژه، Backend تمام قوانین اصلی را کنترول می‌کند. صفحه React فقط درخواست می‌فرستد؛ اما سرور اعتبارسنجی، امنیت، دیتابیس، role و جواب استاندارد را مدیریت می‌کند.»

درس ۱

معماری Modular Monolith یعنی چه؟

DocChain یک modular monolith است. یعنی پروژه به چند microservice جدا تقسیم نشده؛ همه APIها در یک Express server اجرا می‌شوند. اما کدها بر اساس feature جدا شده‌اند: auth، documents، templates، signatures، workspace، admin و غیره.

مزیت برای پروژه پوهنتونی

راه‌اندازی ساده، یک دیتابیس مرکزی، debugging آسان‌تر، و فهم بهتر برای تیم کوچک.

مزیت برای code organization

هر بخش route و helper خودش را دارد؛ پس کدها با وجود یک سرور، پراکنده و بی‌نظم نیستند.

جمله دفاعی

«ما modular monolith را انتخاب کردیم چون سیستم واحد اداری است، اما هر feature در module جدا نگهداری می‌شود تا هم ساده deploy شود و هم code قابل دفاع باشد.»

درس ۳

Middleware Chain: درخواست قبل از route از چه مسیر می‌گذرد؟

Middleware مثل checkpointهای امنیتی و فنی است. هر درخواست قبل از رسیدن به route اصلی، از چند مرحله می‌گذرد.

pinoHttp

درخواست‌ها را log می‌کند تا debugging و audit فنی آسان شود.

compression

جواب‌ها را فشرده می‌کند تا انتقال سریع‌تر باشد.

helmet

headerهای امنیتی HTTP را اضافه می‌کند.

cors

مشخص می‌کند فرانت‌اند از کدام origin اجازه درخواست دارد.

express.json

JSON body را می‌خواند و داخل request.body می‌گذارد.

session

شناسه کاربر، active assignment و CSRF token را در session نگه می‌دارد.

جمله دفاعی

«قبل از اجرای route، request از middlewareها عبور می‌کند؛ این کار باعث می‌شود logging، امنیت، session و parsing به شکل مرکزی مدیریت شود، نه اینکه در هر route تکرار شود.»

POST /api/auth/login -> validate body with Zod -> find user by email or username -> check account lock/status -> argon2.verify(password) -> regenerate session -> save userId and activeAssignmentId -> create CSRF token -> write audit log -> return AuthSession

درس ۵

CSRF، Authentication و Role Protection

requireAuth

بررسی می‌کند session کاربر معتبر است و user هنوز active است.

Open requireAuth

requireAnyRole

برای routeهای admin بررسی می‌کند role کاربر اجازه دسترسی دارد یا نه.

Open role guard

csrfGuard

برای درخواست‌های تغییر‌دهنده مثل POST، PATCH و DELETE token را بررسی می‌کند.

Open csrfGuard

مسیرهای login، forgot password و reset password از CSRF معاف‌اند، چون کاربر هنوز session معتبر ندارد. مسیر phone signature upload هم جداست، چون با token عمومی کار می‌کند.
