سوال و جواب دفاع - Admin و Database

این سوال‌ها برای تمرین شفاهی احمد زوبین است. هر جواب اول کوتاه گفته می‌شود، بعد اگر استاد توضیح بیشتر خواست، جواب عمیق‌تر ارائه می‌شود.

Architecture و نقش Admin

۱. بخش Admin در DocChain چه وظیفه دارد؟

جواب کوتاه: ساختار اداری و کاربران را مدیریت می‌کند. جواب عمیق: Admin foundation سیستم است؛ سازمان، واحد، بست، شخص، حساب، نقش و نوع سند را آماده می‌کند تا workflow و امضا بتوانند درست کار کنند.

۲. چرا این بخش را از Staff Workflow جدا ساختید؟

چون Admin کار تنظیمات و master data است، اما Staff Workflow کار روزانه سندهاست. جدا بودن باعث می‌شود صلاحیت‌ها واضح و تغییرات ساختاری کنترل‌شده باشد.

۳. مسیر یک action در صفحه Admin چگونه است؟

صفحه React فرم را می‌گیرد، adminApi درخواست می‌فرستد، route سرور داده را validate می‌کند، query به MySQL اجرا می‌شود و JSON برمی‌گردد.

تا همه صفحه‌های Admin با یک روش به backend وصل شوند. این کار تکرار fetch، query string و مدیریت response را کم می‌کند.

۵. اگر استاد بپرسد «کدام فایل‌ها بخش شماست»، چه می‌گویید؟

صفحه‌های Admin در client/src/pages/admin، API client در client/src/api/admin.ts، routeها در server/src/modules/admin و schema اصلی در server/migrations/sql/001_core_system.up.sql.

Database Model

۶. فرق person و user چیست؟

person هویت واقعی فرد است؛ user حساب login همان فرد است. جدا کردن آن‌ها باعث می‌شود معلومات اداری از معلومات امنیتی جدا بماند.

VS Codeserver/migrations/sql/001_core_system.up.sql:4
۷. چرا assignment لازم است؟

assignment مشخص می‌کند کدام شخص در کدام position فعال است. سندها به وظیفه اداری وابسته‌اند، نه فقط به حساب login.

۸. چرا units حالت hierarchy دارند؟

چون ساختار پوهنتون سلسله‌مراتبی است. فیلد parent_unit_id کمک می‌کند یک دیپارتمنت زیر یک پوهنځی و یک شعبه زیر یک ریاست ذخیره شود.

۹. role با position چه فرق دارد؟

role صلاحیت امنیتی در سیستم است، مثل admin. position جایگاه اداری است، مثل مدیر دیپارتمنت. یک شخص می‌تواند position داشته باشد ولی role متفاوت بگیرد.

۱۰. چرا foreign key استفاده شده؟

برای جلوگیری از داده یتیم. مثلا assignment بدون person یا position نباید ساخته شود. foreign key همین رابطه را در سطح دیتابیس محافظت می‌کند.

۱۱. چرا بعضی جدول‌ها status دارند؟

برای غیرفعال کردن نرم اطلاعات بدون حذف فوری. مثلا user یا unit می‌تواند inactive شود اما تاریخچه سندها محفوظ بماند.

۱۲. چرا unique index روی codeها مهم است؟

کد سازمان، واحد یا بست باید تکراری نباشد تا سیستم هنگام جستجو، import و reference دادن دچار ابهام نشود.

ساخت و ویرایش organization و unit، نمایش hierarchy، assign head، export/import structure و حذف کنترل‌شده واحدها را انجام می‌دهد.

۱۴. صفحه Users چرا اول person می‌سازد؟

چون user به person وصل است. بدون person، حساب login هویت واقعی و معلومات تماس ندارد.

۱۵. صفحه Positions چه مفهوم مهمی دارد؟

بست‌ها صلاحیت اداری را نشان می‌دهند. فیلدهایی مثل authority_level و is_signing_authority در workflow و signature مهم می‌شوند.

۱۶. صفحه Assignments چه مشکل واقعی را حل می‌کند؟

تغییرات وظیفه‌ای کارمندان را مدل می‌کند. یک شخص ممکن است امروز در یک بست و فردا در بست دیگر باشد؛ assignment این تاریخچه را قابل مدیریت می‌کند.

۱۷. Document Types به کدام بخش‌ها وصل است؟

به template binding، create document، serial rule و workflow. نوع سند تعیین می‌کند قالب و شماره‌دهی چگونه انتخاب شود.

۱۹. چرا بعضی routeها role لازم دارند؟

چون همه کاربران نباید بتوانند ساختار اداری یا نوع سند را تغییر دهند. role check جلوی دسترسی غیرمجاز را می‌گیرد.

۲۰. validation در backend چرا لازم است وقتی frontend فرم دارد؟

Frontend قابل دور زدن است. backend باید همیشه داده را دوباره بررسی کند تا درخواست خراب یا مخرب وارد دیتابیس نشود.

۲۱. audit log در Admin چه فایده دارد؟

تغییرات حساس مثل ساخت user، تغییر role یا حذف واحد قابل پیگیری می‌شود. در سیستم اداری accountability مهم است.

۲۲. reset password در صفحه Admin چرا حساس است؟

چون مدیر می‌تواند دسترسی کاربر را تغییر دهد. باید فقط نقش مجاز انجام دهد و بهتر است در audit log ثبت شود.

Migration و Import/Export

۲۳. migration چیست؟

فایل قابل اجرای تغییرات دیتابیس است. با migration، schema روی هر ماشین به شکل یکسان ساخته می‌شود.

۲۴. seed با migration چه فرق دارد؟

migration ساختار جدول‌ها را می‌سازد؛ seed داده اولیه مثل roles، admin user و نمونه‌های ضروری را وارد می‌کند.

۲۵. import structure برای چیست؟

برای وارد کردن ساختار بزرگ سازمانی از فایل workbook. سیستم اول preview می‌سازد تا خطاها دیده شود، بعد apply انجام می‌شود.

۲۶. چرا import اول preview دارد؟

چون تغییر ساختار اداری پرخطر است. preview نشان می‌دهد چه recordهایی ایجاد یا update می‌شوند و خطاها قبل از ذخیره دیده می‌شود.

Demo و Troubleshooting

۲۷. برای demo سریع چه چیزی نشان بدهید؟

یک unit یا user بسازید و بگویید frontend درخواست را به adminApi می‌دهد، backend route آن را ذخیره می‌کند و جدول مربوط update می‌شود.

۲۸. اگر ساخت user خطا بدهد، کجا را بررسی می‌کنید؟

اول فرم و console، بعد adminApi، بعد route /api/admin/users، بعد unique constraintهای email و username در دیتابیس.

۲۹. اگر یک unit حذف نشود، دلیل احتمالی چیست؟

ممکن است unit فرزند، position یا document وابسته داشته باشد. foreign key یا منطق backend جلوی حذف خطرناک را می‌گیرد.

۳۰. جمع‌بندی ۳۰ ثانیه‌ای شما چیست؟

«Admin و Database پایه، هویت اداری سیستم را می‌سازد. organizations و units ساختار پوهنتون اند؛ positions بست‌ها اند؛ persons افراد واقعی اند؛ users حساب ورود اند؛ assignments شخص را به وظیفه وصل می‌کند. این اطلاعات از صفحه‌های Admin به API، route و MySQL می‌رود.»
