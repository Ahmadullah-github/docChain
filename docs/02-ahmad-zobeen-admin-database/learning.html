مینی کورس دفاع: Admin Panel و دیتابیس پایه

احمد زوبین مسؤول توسعه و دفاع این بخش است. بخش مربوط به این عضو تیم، ستون فقرات اداری DocChain را توضیح می‌دهد: ساختار پوهنتون، اشخاص، کاربران، بست‌ها، وظیفه‌ها، نوع سندها و رابطه‌های اصلی دیتابیس.

جمله کلیدی برای شروع دفاع: «من بخش Admin و دیتابیس پایه را تشریح می‌کنم؛ بدون این بخش سیستم نمی‌داند کاربر کی است، در کدام واحد کار می‌کند، چه صلاحیت دارد، و سند مربوط کدام ساختار اداری است.»

۱. هدف Admin Panel

Admin Panel برای کارهای روزمره کارمند نیست؛ برای آماده‌سازی و مدیریت بنیاد سیستم است. مدیر در این بخش پوهنتون، واحدها، کاربران، بست‌ها و نقش‌ها را تعریف می‌کند تا بخش‌های دیگر بتوانند workflow، امضا و قالب را درست استفاده کنند.

۲. چرا دیتابیس مهم است؟

دیتابیس فقط محل ذخیره نیست؛ قوانین رابطه را هم نگه می‌دارد. برای مثال، یک user باید به یک person وصل باشد و یک position باید داخل یک unit قرار داشته باشد.

۳. چرا این بخش دفاع مهم دارد؟

استادان معمولا از همین بخش سوال رابطه‌ای می‌پرسند: چرا person جداست؟ چرا assignment داریم؟ چرا foreign key استفاده شده؟ اگر این بخش خوب فهمیده شود، کل پروژه قابل توضیح می‌شود.

لکچر ۱: مدل ذهنی ساختار اداری

DocChain مثل یک اداره واقعی فکر می‌کند. اول یک سازمان داریم، بعد داخل آن واحدها قرار می‌گیرند، داخل هر واحد بست‌ها تعریف می‌شوند، و اشخاص از طریق assignment به بست‌ها وصل می‌شوند.

organizations -> unit_types -> units -> positions -> assignments -> persons -> users

نکته برای فهم

اگر یک استاد بپرسد «کاربر کجا کار می‌کند؟» جواب از جدول users مستقیم نمی‌آید؛ از مسیر person -> assignment -> position -> unit پیدا می‌شود.

روش گفتن به استاد

«ما حساب login را از هویت اداری جدا کردیم تا یک شخص بتواند معلومات واقعی، نقش امنیتی و وظیفه اداری جداگانه داشته باشد.»

لکچر ۲: مسیر درخواست از صفحه تا دیتابیس

وقتی مدیر یک کاربر یا واحد جدید می‌سازد، جریان به این شکل است: صفحه React فرم را می‌گیرد، adminApi درخواست HTTP می‌فرستد، route سرور داده را validate می‌کند، SQL اجرا می‌شود، و نتیجه دوباره به صفحه برمی‌گردد.

Admin page -> adminApi -> /api/admin/... -> admin.routes.ts -> MySQL -> JSON response

مثال ساخت کاربر

صفحه AdminUsersPage.tsx اول person می‌سازد، بعد user را به همان person وصل می‌کند، سپس نقش‌ها و assignment را مدیریت می‌کند.

روش گفتن به استاد

«Frontend فقط نمایش و جمع‌آوری معلومات را انجام می‌دهد؛ تصمیم اصلی و ذخیره نهایی در backend route و MySQL انجام می‌شود.»

لکچر ۳: Organizations و Units

سازمان سطح کلان است. واحدها داخل سازمان می‌آیند و می‌توانند parent داشته باشند؛ یعنی واحد زیرمجموعه واحد دیگر باشد. این ساختار برای پوهنتون طبیعی است: ریاست، پوهنځی، دیپارتمنت و شعبه.

  • جدول organizations نام و کد سازمان را نگه می‌دارد.
  • جدول unit_types نوع واحد را تعریف می‌کند.
  • جدول units با parent_unit_id سلسله‌مراتب را می‌سازد.

لکچر ۴: Persons و Users

person هویت واقعی فرد است، اما user حساب ورود به سیستم است. این جداسازی باعث می‌شود معلومات اداری مثل نام و شماره تماس با معلومات امنیتی مثل password hash و status حساب مخلوط نشود.

  • persons: نام، ایمیل، شماره تماس، وضعیت فرد.
  • users: username، password hash، lockout، last login.
  • user_roles: وصل کردن کاربر به نقش‌های امنیتی.

لکچر ۵: Positions و Assignments

بست یک جایگاه اداری است؛ assignment یعنی کدام شخص فعلا آن بست را گرفته است. این طراحی کمک می‌کند اگر یک شخص تبدیل شود، تاریخچه و رابطه‌های قبلی از بین نروند.

فیلدهایی مثل is_signing_authority و authority_level روی position برای صلاحیت امضا و سلسله اداری استفاده می‌شوند.

لکچر ۶: Document Types و Write Rules

نوع سند مشخص می‌کند سند از چه نوع است، مثل مکتوب، تصدیق یا فورم. Write rules مشخص می‌کند کدام واحد یا بست حق ایجاد همان نوع سند را دارد. این بخش با workflow و templates رابطه مستقیم دارد.

document_types -> document_write_rules -> templates/workflow/signature

لکچر ۷: Migration و Seed

Migration فایل SQL است که ساختار دیتابیس را قابل تکرار می‌سازد. وقتی پروژه روی کمپیوتر جدید نصب شود، migration همان جدول‌ها، foreign keyها و indexها را می‌سازد. Seed داده اولیه مثل نقش‌ها، سازمان نمونه و کاربر ادمین را وارد می‌کند.

چرا migration خوب است؟

چون ساختار دیتابیس از ذهن برنامه‌نویس بیرون می‌آید و داخل فایل قابل اجرا و قابل بررسی قرار می‌گیرد.

روش گفتن به استاد

«اگر دیتابیس پاک شود یا روی ماشین دیگر نصب شود، migrationها دوباره همان schema را می‌سازند و seed داده‌های ضروری را اضافه می‌کند.»

چک‌لیست مطالعه احمد زوبین

بتواند فرق person و user را توضیح دهد.
بتواند مسیر organization -> unit -> position -> assignment را روی تخته رسم کند.
بتواند بگوید چرا foreign key برای صحت رابطه‌ها ضروری است.
بتواند یک demo کوتاه ساخت user یا unit را همراه مسیر کد توضیح دهد.
بتواند صفحه‌های Admin را با routeهای backend وصل کند.
بتواند بگوید document type و write rule چرا برای workflow لازم است.

تمرین دفاع شفاهی
