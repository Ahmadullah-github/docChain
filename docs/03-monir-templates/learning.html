مینی کورس دفاع: Templates، Builder و Rendering

منیر مسؤول توسعه و دفاع این بخش است. بخش مربوط به این عضو تیم تعیین می‌کند سند رسمی چگونه دیده شود: لوگو، سربرگ، متن، جدول، امضا، QR، فاصله‌ها، صفحه A4، HTML preview و PDF نهایی.

جمله شروع دفاع: «در DocChain محتوا از ظاهر جدا است. کارمند محتوا را وارد می‌کند، اما template تصمیم می‌گیرد سند رسمی چگونه نمایش، چاپ و PDF شود.»

۱. Template چیست؟

Template نقشه ظاهری سند است. مثل فورم آماده‌ای که مشخص می‌کند عنوان کجا باشد، متن در کدام بخش بیاید، امضاها کجا قرار گیرند و QR در کدام گوشه چاپ شود.

۲. چرا Builder لازم است؟

اگر قالب‌ها hard-code شوند، هر تغییر ظاهری نیاز به برنامه‌نویس دارد. Builder به ادمین اجازه می‌دهد قالب رسمی را با ابزار گرافیکی تنظیم کند.

۳. Renderer چه کار می‌کند؟

Renderer تعریف JSON قالب و معلومات سند را می‌گیرد و خروجی HTML رسمی تولید می‌کند؛ بعد PDF renderer همان HTML را به فایل قابل چاپ تبدیل می‌کند.

لکچر ۱: lifecycle کامل قالب

Create template -> Edit layout -> Save version -> Submit -> Review -> Approve -> Bind -> Preview -> Render HTML/PDF

قالب فقط یک صفحه طراحی نیست؛ یک جریان کامل دارد. کاربر قالب را می‌سازد، نسخه قابل ویرایش ذخیره می‌شود، برای تایید ارسال می‌شود، ادمین آن را approve می‌کند، بعد به نوع سند bind می‌شود تا در ساخت سند قابل انتخاب باشد.

روش گفتن به استاد: «ما قالب را versioned ساختیم تا نسخه‌های قبلی از بین نروند و قالب رسمی بدون review مستقیم فعال نشود.»

لکچر ۲: Template Library

Template Library جای دیدن قالب‌های آماده و قابل clone است. کاربر می‌تواند قالب موجود را بگیرد، clone کند و برای نوع سند خاص تنظیم نماید. این کار سرعت ساخت قالب را بالا می‌برد.

در دفاع بگویید library برای reuse است؛ یعنی لازم نیست هر قالب از صفر ساخته شود.

لکچر ۳: Template Builder

Builder از چند قسمت تشکیل شده: canvas برای دیدن صفحه، ribbon برای ابزارها، rail یا inspector برای تنظیمات، و model functions برای ساخت layout سالم. نتیجه Builder یک object ساختاری به نام TemplateLayout است.

لکچر ۴: Blocks و Tokens

قالب از blockها ساخته می‌شود: متن، تصویر، جدول، خط، QR، امضا و fieldهای متحرک. Tokenها جای‌نگهدارهایی مثل شماره سند، تاریخ، نام واحد و امضای تکمیل‌شده هستند.

{{document.subject}}, {{document.serial}}, {{signature.completed}}, {{origin_unit.name}}

لکچر ۵: Preview و A4

Preview کمک می‌کند ادمین قبل از ذخیره یا چاپ، ظاهر سند را ببیند. صفحه باید مثل A4 رفتار کند، جهت متن را رعایت کند، و فاصله‌ها در HTML و PDF نزدیک به هم باشند.

لکچر ۶: Approval و Binding

وقتی قالب submit شد، در review queue می‌آید. ادمین می‌تواند approve یا reject کند. بعد قالب approved به document type bind می‌شود؛ یعنی سیستم می‌فهمد برای آن نوع سند کدام قالب فعال است.

لکچر ۷: Render HTML و PDF

Render مرحله‌ای است که layout و document data یکجا می‌شوند. HTML برای preview و نمایش رسمی استفاده می‌شود؛ PDF برای دانلود، چاپ و آرشیف رسمی مناسب‌تر است.

دیتابیس Templates

document_templates

معلومات اصلی قالب، owner، status، visibility و current version را نگه می‌دارد.

document_template_versions

نسخه‌های layout، submit/review status و تاریخچه تغییرات را نگه می‌دارد.

document_template_bindings

وصل کردن قالب approved به یک document type یا context خاص.

document_layout_drafts

draft layout سندهای کارمندان را ذخیره می‌کند تا preview/render بعدی از بین نرود.

چک‌لیست مطالعه منیر

فرق content سند و layout قالب را توضیح دهد.
lifecycle قالب را از create تا binding روی تخته رسم کند.
بداند TemplateLayout و blockها چه مفهومی دارند.
بتواند renderer HTML و PDF renderer را از هم جدا توضیح دهد.
بتواند علت versioning و approval را دفاع کند.
بتواند ارتباط قالب با document type و staff create page را توضیح دهد.

تمرین دفاع شفاهی

«من بخش Templates را توضیح می‌دهم. هدف ما جدا کردن محتوای سند از شکل ظاهری است. قالب در Builder ساخته می‌شود، به شکل version ذخیره می‌شود، برای review ارسال می‌شود، بعد approve و به document type وصل می‌گردد. هنگام ساخت یا نمایش سند، renderer معلومات سند را با layout ترکیب کرده HTML و PDF رسمی تولید می‌کند.»
